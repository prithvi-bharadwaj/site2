## Problem Statement

Build a personal portfolio site where the entire visual surface is a single, unified particle system. Every piece of content, name, tagline, navigation, project descriptions, images, is rendered as particles on one full-viewport WebGL canvas. The cursor interacts with the particle field as one living system. The site should be instantly impressive to non-technical viewers, technically respectable to creative coders, and readable enough to function as a real portfolio.

## What Makes This Cool

Most developer portfolios put a flashy hero at the top and then become a normal webpage. This one never breaks the illusion. The particles ARE the site. Scroll and content dissolves and reforms. Move your mouse and the whole field reacts. It's soulwire-level generative art that also happens to be a readable portfolio. The constraint, readable particle text at scale, is what makes it hard and interesting.

## Constraints

- Must run at 60fps on mobile (iPhone 14 / Pixel 7 class)
- Text must be genuinely readable, not "art first, squint later"
- No visually rendered HTML content. All visible content is particle-rendered. Hidden HTML elements exist for accessibility (screen reader mirror, keyboard-navigable links/buttons).
- WebGL 2 for broad support (98%+ browser coverage)
- Particle count: desktop 8,000-12,000 / mobile 3,000-5,000
- Existing particle-text.ts physics/morphing logic should be reused (the math is solid)
- Single-page feel with scroll-driven content transitions
- Reduced motion fallback (static rendered text, no animation)
- Currently deployed on Vercel

## Premises

1. One full-viewport particle system renders ALL visible content. Hidden HTML for accessibility only.
2. Particle text must be readable. Higher density, sharper rendering for body text.
3. WebGL-based renderer (instanced quads) for GPU-accelerated particles at 8,000-12,000 (desktop) / 3,000-5,000 (mobile).
4. The particle-text.ts physics/morphing logic ports to Float32Array struct-of-arrays layout. The sampling functions (textToPoints, imageToPoints) are reusable as-is. The physics functions (morphTo, updatePhysics) require data layout migration from Particle[] objects to interleaved Float32Arrays (14 floats per particle). The rendering layer moves to WebGL.

## Approaches Considered

### Approach A: WebGL Point Sprites
One full-viewport WebGL canvas. All particles are point sprites rendered via gl.POINTS with a custom fragment shader for glow. Physics runs on CPU in typed arrays.

Effort: M (human: ~1 week / CC: ~45 min)
Risk: Low
Pros: Simplest WebGL path, fewest shader lines, easy point size control
Cons: Point sprite max size varies by GPU (64-256px), circles only

### Approach B: WebGL Instanced Quads (CHOSEN)
Same architecture but particles are instanced quads with a radial gradient texture. Each quad can be any size, supports glow halos, and scales beyond point sprite limits.

Effort: L (human: ~2 weeks / CC: ~1.5 hours)
Risk: Medium
Pros: No point size limits, per-particle glow halos in shader, better visual quality
Cons: More boilerplate (vertex buffer, instance buffer, index buffer), slightly more complex shader

### Approach C: GPU Compute Particles (WebGPU)
Physics AND rendering on the GPU via WebGPU compute shaders.

Effort: XL (human: ~1 month / CC: ~3 hours)
Risk: High (no Safari iOS)
Rejected: Overkill for 10k particles and kills mobile support.

## Recommended Approach

**Approach B: WebGL Instanced Quads.** The glow halo capability is load-bearing for the "must be readable" constraint. Each particle renders as an instanced quad with a radial gradient fragment shader (pre-baked 32x32 texture lookup for mobile GPU performance), giving crisp cores with soft halos. Physics runs on CPU in Float32Arrays (12k particles is trivial for modern CPUs). The existing textToPoints/imageToPoints/samplePixels functions reuse as-is. The physics functions (morphTo, updatePhysics, updateMorph) require migration from Particle[] object arrays to interleaved Float32Arrays (14 floats per particle: x, y, homeX, homeY, vx, vy, originX, originY, targetX, targetY, cpX, cpY, size, alpha). This is a data layout change, not an algorithm change.

### Architecture

```
  ┌─────────────────────────────────────────────────────────────┐
  │                    Full-Viewport WebGL Canvas                │
  │                                                              │
  │  ┌──────────┐   ┌──────────┐   ┌──────────┐                │
  │  │ Region:  │   │ Region:  │   │ Region:  │                │
  │  │  Hero    │   │  Nav     │   │ Projects │ ...             │
  │  │ (name,  │   │ (links)  │   │ (cards)  │                │
  │  │ tagline) │   │          │   │          │                │
  │  └──────────┘   └──────────┘   └──────────┘                │
  │                                                              │
  │  All regions share ONE particle pool.                        │
  │  Scroll position determines which region particles morph to. │
  └──────────────────────────────────────────────────────────────┘

  DATA FLOW:
  Scroll position ──> Region visibility ──> morphTo(region targets)
  Mouse position ──> Physics repulsion (global, all particles)
  RAF loop ──> updatePhysics() ──> WebGL instanced draw()
```

### Key Components

1. **ParticleEngine** (new) — Singleton. Owns the Float32Array particle pool, runs physics, manages regions.
2. **WebGLRenderer** (new) — Instanced quad renderer. Vertex shader positions quads per particle. Fragment shader draws radial gradient with configurable glow.
3. **Region** (new) — Defines a content area with: target points (from text/image sampling), a scroll range `[startY, endY]` in page pixels where the region is active, a particle slice (index range into the shared pool), and a density multiplier. Regions are vertically stacked. Each region owns a contiguous slice of the particle pool: hero gets indices 0..2999, nav gets 3000..3499, projects get 3500..9499, footer gets 9500..11999 (desktop). On mobile, slices scale proportionally with the reduced pool.
4. **Existing reuse** — textToPoints, imageToPoints, samplePixels reuse as-is. morphTo, updateMorph, updatePhysics port with Float32Array data layout change (same algorithms, different memory layout).

### Scroll Behavior

Uses `window.scrollY` directly (not IntersectionObserver) to get continuous `t` values for smooth morphing. Each region has a `[startY, endY]` scroll range. The normalized progress `t = (scrollY - startY) / (endY - startY)` drives morph interpolation.

**Morph + physics coexistence:** The existing code treats morph and spring physics as mutually exclusive states (`animState: "morphing" | "idle"`). For scroll-driven transitions, this changes: during a morph (`t` between 0 and 1), particle positions are set by bezier interpolation (morph dominates). Mouse repulsion applies as a displacement offset on top of the morph position, not as an alternative state. When `t` reaches 1 (region fully visible, idle), pure spring physics takes over. This means `updateMorph` no longer zeroes velocity, it preserves `vx/vy` so the transition from morph to physics is seamless.

As user scrolls:
- Particles in the hero region dissolve (scatter slightly or fade alpha via `t`)
- Particles morph to the next region's target positions along bezier curves
- During overlap transitions (two regions partially visible), each region's particle slice morphs independently, each with its own `t` value derived from scroll position
- Mouse repulsion is always active, applied as offset displacement during morph, spring-based during idle
- If `morphTo()` is called on a slice already mid-morph (rapid scrolling), it snaps origin to current position and sets new target. This is the existing behavior and is acceptable.
- Creates continuous flow, not page transitions

Scroll listener runs on `requestAnimationFrame` (not raw scroll events) to batch with the render loop. No jank.

### Pool Allocation Strategy

```
  DESKTOP (12,000 particles):

  │  Hero    │ Nav  │  Projects    │ Footer │
  │  3,000   │ 500  │  6,000       │ 2,500  │
  │  (name,  │      │  (4 cards    │ (flower │
  │  tagline)│      │  @ 1,500 ea) │  image) │
  └──────────┴──────┴──────────────┴────────┘

  MOBILE (4,000 particles):
  ┌──────────┬──────┬──────────────┬────────┐
  │  Hero    │ Nav  │  Projects    │ Footer │
  │  1,200   │ 200  │  2,000       │  600   │
  └──────────┴──────┴──────────────┴────────┘
```

Each region owns a contiguous slice of the Float32Array. No allocation conflicts during scroll transitions because slices don't overlap.

### Resize Handling

On `window.resize` (debounced 200ms):
1. Recalculate canvas dimensions (full viewport)
2. Re-run textToPoints/imageToPoints for affected regions (text metrics change with viewport width)
3. Update region scroll ranges (content height may change)
4. Morph existing particles to new target positions (no pool reallocation, just new targets)
5. Recalculate mobile/desktop particle count if breakpoint crossed (768px). If crossing breakpoint, reallocate pool.

### Navigation

Hidden `<nav>` with real `<a>` tags for accessibility and keyboard navigation. These elements are positioned using `position: fixed` and transformed to track the canvas coordinate of their particle region via a `syncHitTargets()` function called each frame. `opacity: 0`, `pointer-events: auto`. Screen readers and keyboard users get real links. Visual users see the particle-rendered nav. Click handlers trigger `window.scrollTo({ behavior: 'smooth' })` to the target region's `startY`.

### Project Card Interaction

**V1 (ship first):** Each project card in the particle grid is a hidden `<a>` link. Clicking navigates to the project URL (external link) or scrolls to expand. No particle-morph modal in v1.

**V2 (stretch goal):** Particle-morph expand/collapse. Click scatters the card's particle slice outward, then reforms as expanded detail view. This is a separate feature with its own implementation scope, not part of the initial build.

### Accessibility

- Hidden HTML mirror: `<main>` with all content text in `sr-only` (visually-hidden) elements for screen readers
- `<canvas>` has `role="img"` and `aria-label` with full page description
- Keyboard navigation via hidden `<a>` and `<button>` elements, position-synced each frame via `syncHitTargets()`
- Focus bridge: hidden elements emit `focus`/`blur` events. The React component listens via `onFocus`/`onBlur` and sets a `focusedRegion` state on the engine, which triggers a subtle glow pulse on that region's particle slice (increased `u_glowIntensity` uniform for that region's draw call)
- Reduced motion: static particles at final positions (no animation, no physics, no morph). NOT HTML fallback. Particles still render, just frozen.

### Loading / Initialization

1. Page loads with a black screen (matches `#0a0a0a` background)
2. WebGL context created, shaders compiled, buffers allocated
3. Regions initialize sequentially across multiple RAFs to avoid jank: hero first (frame 1), nav (frame 2), projects (frame 3-4)
4. As each region's `textToPoints` completes, its particles scatter-in from random positions
5. Total init time target: < 500ms on desktop, < 1000ms on mobile
6. If WebGL context creation fails: fall back to static HTML (the hidden accessibility mirror becomes visible)

### Engine Lifecycle

`ParticleEngine` is a singleton created in the root page component's `useEffect`. It owns:
- The Float32Array particle pool
- The WebGL context and buffers
- The RAF loop
- Scroll and mouse listeners

On unmount (Next.js navigation away): cancel RAF, remove listeners, delete WebGL resources (`gl.deleteBuffer`, `gl.deleteProgram`, `gl.deleteTexture`). On `webglcontextlost`: pause RAF, set `contextLost` flag. On `webglcontextrestored`: recreate all WebGL resources, re-upload buffers, resume RAF. In Next.js dev mode (hot reload): full teardown and re-init via useEffect cleanup.

### Mobile Strategy

- Reduce particle count to 4,000 on mobile (window.innerWidth < 768)
- Lower WebGL resolution (0.75x DPR)
- Reduce quad size via `u_sizeMultiplier` uniform (same shader, smaller particles)
- Touch: repulsion on touch-move, no hover state

### Fragment Shader & Mobile Glow

Desktop: fragment shader samples a pre-baked 32x32 radial gradient texture for the glow halo. Single draw pass with the glow baked into the texture (the texture itself has a soft falloff, no separate glow pass needed).

Mobile: same single draw pass but with a smaller quad scale (`u_sizeMultiplier` uniform reduced). The texture handles the visual, the uniform controls the size. One shader, no branching.

## Resolved Decisions

1. **Color:** All white particles. Per-particle color adds 3 floats to the instance buffer (14 -> 17 floats/particle, +21% buffer size) and complicates the shader for minimal visual gain. The accent purple for the name can be achieved with a per-region `u_color` uniform instead, zero buffer cost. Float32Array layout stays at 14 floats/particle.
2. **Writing section:** Excluded from the particle site for v1. Writing entries have long text that would consume too many particles. The `/writing` route stays as a separate HTML page using the existing PageShell component. This can be revisited in v2.
3. **Pool allocation (fixed):** Desktop 12,000 / Mobile 4,000. No range, single value.

## Open Questions

1. Should rapid-scroll morph interruption use "snap to current position and re-morph" (current `morphTo` behavior) or "blend between in-flight and new target"? Current behavior is acceptable but could feel jumpy at high scroll speeds.

## Known Risks

- **iOS Safari WebGL 2 instanced rendering:** `ANGLE_instanced_arrays` has had driver quirks historically. Safari 16+ (2022) is solid, but older devices may fail. Fallback: detect `gl.drawElementsInstanced` at init. If unavailable, fall back to non-instanced draw calls (slower but functional) or static HTML.
- **WebGL context loss:** GPU resets, tab backgrounding on iOS can lose the context. Handle via `webglcontextlost` / `webglcontextrestored` events. On loss: pause RAF. On restore: recreate buffers and textures, re-upload particle data.
- **Breakpoint resize buffer reallocation:** Crossing the 768px breakpoint requires reallocating the Float32Array and re-uploading the WebGL instance buffer (~672KB at 12k particles). This is a one-time GPU upload that may cause a single-frame stutter. Mitigation: defer reallocation to next RAF, show particles at current count until upload completes.
- **textToPoints synchronous DOM work:** Multiple regions calling `document.createElement("canvas")` at init creates a jank spike. Mitigation: stagger region initialization across multiple RAFs (hero first, then nav, then projects on subsequent frames).

## Success Criteria

- 60fps on iPhone 14 Safari (4000 particles, single glow pass)
- 60fps on Chrome desktop (12000 particles, dual glow pass)
- Body text readable: minimum 40 particles per glyph at 14px font size, particle diameter >= 1.5px
- Mouse repulsion feels responsive (< 16ms latency)
- Total JS bundle < 50KB gzipped (WebGL renderer + physics)
- Lighthouse performance score > 90
- First meaningful paint (hero particles visible) < 500ms on desktop
- WebGL context loss recovery without page reload
